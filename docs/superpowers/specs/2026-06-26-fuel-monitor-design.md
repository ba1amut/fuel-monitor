# Fuel Monitor — Дизайн-документ

**Дата:** 2026-06-26  
**Статус:** Согласован  
**Проект:** Краудсорсинговый мониторинг наличия топлива на АЗС России

---

## Контекст и цель

Социальный проект для агрегации информации от очевидцев о ситуации с топливом на АЗС по всей России. Пользователи сообщают о наличии/отсутствии топлива, марках и ценах через Telegram-бот. Система аккумулирует данные и предоставляет сводку по запросу через бот и публичную веб-карту.

---

## Фаза 0 — Технический спайк (до основной разработки)

Перед полной реализацией необходимо подтвердить работоспособность ключевого пайплайна в текущих условиях (ранее наблюдались ограничения со стороны Telegram).

**Цель спайка:** минимальный Telegram-бот, который:
1. Принимает текстовое сообщение → передаёт в YandexGPT → возвращает распознанный результат
2. Принимает фото → передаёт в YandexGPT Vision → возвращает результат
3. Принимает голосовое сообщение → транскрибирует через Yandex SpeechKit → передаёт в YandexGPT → возвращает результат

Только после успешного прохождения всех трёх сценариев переходим к Фазе 1.

## Архитектура

**Стек:** Python-монолит — FastAPI + aiogram 3 + PostgreSQL/PostGIS + Leaflet.js  
**Деплой:** Docker Compose на FirstVDS, Nginx как reverse proxy, SSL через Let's Encrypt

```
┌─────────────────────────────────────────────────┐
│                  Docker Compose                  │
│                                                  │
│  ┌──────────┐   ┌──────────┐   ┌─────────────┐  │
│  │ Telegram │   │  FastAPI │   │  PostgreSQL  │  │
│  │   Bot    │──▶│   API    │──▶│  + PostGIS  │  │
│  │ (aiogram)│   │          │◀──│             │  │
│  └──────────┘   └────┬─────┘   └─────────────┘  │
│  текст/фото/голос    │                           │
│                ┌─────▼──────┐                    │
│                │ Static Web │                    │
│                │ (Leaflet)  │                    │
│                └────────────┘                    │
└─────────────────────────────────────────────────┘
```

**Пайплайн входящих сообщений:**

```
Текст  ──────────────────────────────▶ YandexGPT (парсинг)
Фото   ──▶ base64 ───────────────────▶ YandexGPT Vision
Голос  ──▶ OGG файл ──▶ SpeechKit ───▶ текст ──▶ YandexGPT (парсинг)
```

**Компоненты:**

1. **Telegram-бот (aiogram 3)** — транспортный слой. Принимает текст, фото и голосовые сообщения, передаёт в API, возвращает ответы. Бизнес-логики не содержит.
2. **FastAPI** — центральная логика. Принимает отчёты, вызывает Yandex Cloud для парсинга, сохраняет в БД, отвечает на запросы сводки. Раздаёт статику веб-карты.
3. **PostgreSQL + PostGIS** — хранение отчётов с гео-данными. PostGIS обеспечивает гео-запросы (радиус, агрегация по регионам).
4. **Веб-карта (Leaflet.js)** — публичный дашборд. Маркеры АЗС с кластеризацией + тепловой слой по регионам.

**Структура проекта:**

## Yandex Cloud — используемые сервисы

Один аккаунт, один `YANDEX_FOLDER_ID`, два сервиса:

| Сервис | Назначение |
|---|---|
| **YandexGPT** | Парсинг текстовых отчётов — извлечение бренда, названия АЗС, марок топлива, цен |
| **YandexGPT Vision** | Парсинг фотографий табло цен |
| **Yandex SpeechKit** | Транскрипция голосовых сообщений (OGG/OPUS → текст) |

**Переменные окружения:**
```
TELEGRAM_TOKEN=
YANDEX_API_KEY=
YANDEX_FOLDER_ID=
DATABASE_URL=
```

## Модель данных

### `users` — участники системы
```sql
users
├── telegram_user_id  BIGINT, PK
├── report_count      INT
├── first_seen_at     TIMESTAMPTZ
├── last_seen_at      TIMESTAMPTZ
└── is_banned         BOOLEAN     -- ручная блокировка при вбросах
```

### `stations` — справочник АЗС (накапливается из отчётов)
```sql
stations
├── id              UUID, PK
├── brand           TEXT        -- «Лукойл», «Роснефть», «независимая»…
├── aliases         JSONB       -- ["АЗС №47", "Лукойл на Ленинском", "у Меги"]
├── address         TEXT
├── location        GEOMETRY(Point, 4326)
├── city            TEXT
├── region          TEXT        -- субъект РФ
├── last_report_at  TIMESTAMPTZ
└── report_count    INT
```

### `reports` — все отчёты очевидцев
```sql
reports
├── id                UUID, PK
├── station_id        UUID, FK → stations.id
├── telegram_user_id  BIGINT, FK → users.telegram_user_id
├── raw_text          TEXT              -- оригинальный текст, транскрипт голоса или описание фото
├── has_photo         BOOLEAN
├── fuels             JSONB             -- [{"grade":"АИ-95","available":true,"price":78.5}]
├── user_location     GEOMETRY(Point)   -- геолокация от пользователя, NULL если не передана
├── queue_minutes     INT               -- NULL если не упомянуто
├── confidence        FLOAT             -- уверенность YandexGPT (0–1)
├── parse_failed      BOOLEAN
├── created_at        TIMESTAMPTZ
└── source            TEXT              -- 'telegram_text' | 'telegram_photo' | 'telegram_voice'
```

### `station_fuel_states` — актуальное состояние по каждой марке
```sql
station_fuel_states
├── id              UUID, PK
├── station_id      UUID, FK → stations.id
├── grade           TEXT          -- «АИ-92», «АИ-95», «АИ-100», «ДТ», «ГАЗ»
├── available       BOOLEAN
├── price           NUMERIC(8,2)  -- руб/л, NULL если не указана
├── last_report_id  UUID, FK → reports.id
└── updated_at      TIMESTAMPTZ
```

**Логика обновления:** при новом отчёте — `UPSERT` в `station_fuel_states` для каждой упомянутой марки. Сырые данные остаются в `reports` для истории и модерации.

**Идентификация АЗС:** YandexGPT извлекает бренд и название → нечёткий поиск по `aliases` в пределах города → если найдено: привязываем к существующей станции, новый псевдоним добавляем в массив; если нет: создаём новую запись.

## Telegram-бот: сценарии

### Отправка отчёта
Пользователь пишет в произвольной форме, присылает фото табло цен или голосовое сообщение.

```
Пользователь: «Лукойл на Вернадского, АИ-92 закончился, 95 есть по 79 руб»
              [или фото табло]
              [или голосовое сообщение]

→ Yandex Cloud парсит:
    brand=Лукойл, alias=«на Вернадского»,
    fuels=[{АИ-92: нет}, {АИ-95: есть, 79₽}]
→ Поиск/создание станции в БД
→ UPSERT в station_fuel_states

Бот отвечает:
«Принято! АЗС: Лукойл на Вернадского
 АИ-92: ❌ | АИ-95: ✅ 79₽/л
 Спасибо за помощь 🙏»
```

### Запрос сводки
```
Пользователь: «Что с бензином в Казани?»

Бот отвечает:
«Казань — последние отчёты:

 • Лукойл на Баумана
   АИ-92: ✅ 74₽/л (2 ч назад)
   АИ-95: ✅ 78₽/л (2 ч назад)

 • Роснефть Сибирский тракт
   АИ-92: ❌ (4 ч назад)
   АИ-95: ✅ 82₽/л (4 ч назад)

 • Татнефть Победы
   АИ-92: ✅ 73₽/л (30 мин назад)
   АИ-95: ✅ 77₽/л (30 мин назад)
   ДТ:    ✅ 71₽/л (30 мин назад)»
```

Каждая марка показывает время своего последнего отчёта независимо. Если цена не указана — «✅ (цена не указана)». Марки без ни одного отчёта по данной АЗС не показываются.

### Команды бота

| Команда | Действие |
|---|---|
| `/start` | Приветствие + инструкция |
| `/help` | Как отправить отчёт |
| `/city Казань` | Сводка по городу |
| `/map` | Ссылка на веб-карту |
| Любой текст/фото/голос | Парсинг как отчёт |

### Обработка ошибок парсинга
Если `confidence < 0.5` — бот уточняет: «Не смог разобрать сообщение. Укажи: название АЗС, марку топлива, есть/нет?»

## Веб-карта

**Два слоя:**
- **Маркеры АЗС** с кластеризацией. Цвет: зелёный (есть топливо), красный (дефицит), серый (нет свежих данных). Клик → карточка с детализацией по маркам и временем каждого последнего отчёта.
- **Тепловой слой по регионам** — интенсивность = доля АЗС с дефицитом. Пересчитывается под активную комбинацию фильтров.

**Фильтры (применяются одновременно):**
- По бренду
- По марке топлива

**Обновление:** каждые 5 минут автоматически.

## REST API

```
POST /api/reports              — принять отчёт (текст, фото или голос)
GET  /api/stations             — список АЗС ?brand=&grade=&city=&region=
GET  /api/stations/{id}        — карточка АЗС с детализацией по маркам
GET  /api/heatmap              — агрегация по регионам ?brand=&grade=
GET  /api/summary              — текстовая сводка для бота ?city=&brand=&grade=
```

## Модерация

На старте — доверие сообществу, ручная блокировка через `users.is_banned` при выявлении вбросов. Полная история отчётов с `telegram_user_id` позволяет ретроспективно выявлять и нейтрализовать недобросовестных участников.
